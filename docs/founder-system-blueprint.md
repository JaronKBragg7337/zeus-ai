# Zeus Founder-System Blueprint

Zeus is not only a local AI workbench. The larger goal is a repeatable operating system for creating products, companies, workflows, and real-world systems from scratch.

This repo should document both:

- the software being built
- the method used to build it with AI coworkers

That way another person with a capable local/desktop AI and coding agents can repeat the path instead of only reading final code.

## North Star

Zeus should help a person become a true founder by turning intent into organized execution:

1. capture ideas
2. turn ideas into projects
3. break projects into tasks
4. build the software or process
5. test it
6. package it
7. document it
8. prepare it for presentation and promotion
9. operate the system after launch
10. learn from every action

The user should be able to keep living their life while Zeus keeps the work organized, logged, and moving.

## Repeatable Creation Loop

Every major Zeus workflow should eventually follow this loop:

1. **Intent**
   - What is the user trying to create?
   - What does success look like?

2. **Plan**
   - What has to exist?
   - What can be built now?
   - What needs research, money, compute, or a connector?

3. **Build**
   - Make the smallest working version.
   - Preserve the repo's current structure unless there is a clear reason to change it.
   - Commit in logical steps.

4. **Verify**
   - Run syntax checks, tests, builds, and smoke tests.
   - Record failures and fixes.

5. **Package**
   - Produce something usable: installer, script, doc, dataset, model checkpoint, or service.

6. **Document**
   - Update README/docs.
   - Record commands used.
   - Record known limitations and next steps.

7. **Train**
   - Capture tool traces, corrections, completions, and decisions.
   - Review and clean data before serious model training.

8. **Promote**
   - Prepare screenshots, demos, release notes, landing copy, and user onboarding when the product is ready.

## Systems Zeus Should Grow Toward

- product factory
- company setup assistant
- local file and knowledge operator
- coding and testing operator
- desktop/computer-use operator
- customer research and market map builder
- training-data flywheel
- release and packaging manager
- presentation and promotion assistant
- long-running background project manager

## Why Logging Matters

If the build process is not documented, the system cannot be repeated.

Zeus should record:

- what changed
- why it changed
- commands used
- tests run
- failures and fixes
- training data generated
- design decisions
- remaining limitations

This creates a trail future AI coworkers can follow and improves Zeus-native model training over time.
